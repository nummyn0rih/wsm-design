## 1. Контекст и цели

### 1.1 Продукт

**WSM (Warehouse / Supplier / Weighbridge Management System)** — учёт отгрузок овощного сырья с фермерских хозяйств на перерабатывающий завод. Основные культуры (8): огурцы, томаты, черри, патиссоны, халапеньо, перец острый, перец сладкий, перец сладкий маринованный. Сезонные (6): кабачки, тыква, морковь, свёкла, лук, капуста.

> ⚠ Перец сладкий маринованный приходит на завод уже маринованным от фермера (не процесс на заводе) — полноценный отдельный raw, не подкатегория.

### 1.2 Бизнес-задача

- Прозрачность входящего сырья: что, сколько, от кого, когда поступило (план vs факт).
- Контроль выполнения недельного производственного плана.
- Контроль качества приёмки (брак, нестандарт, калибры) с привязкой к актам.
- Управление оборотом тары (ящики, бочки) между заводом и фермерами.
- Финансовый учёт по контрактам (план vs факт в тоннах и рублях).
- Печатные формы (реестры, сводки) для бухгалтерии и логистов.

### 1.3 Пользователи

- **Админ** (учётчик / завскладом / руководитель смены): все экраны и действия.
- **Оператор** (полевой логист, мобилка): отгрузки + справочник водителей; только базовые операции.
- **Руководитель** (директорский дашборд): будущая роль, отложена.

### 1.4 Скоуп MVP (Phase 1 — Core)

- **Главная страница** с тремя режимами просмотра отгрузок: **Таблица / Heatmap / План**.

- **Левый sidebar** с навигацией, сворачиваемый, с переключателем роли.
- **Role toggle** (мок Admin ↔ User) — без реальной авторизации.
- **Справочник «Водители»** (просмотр для User, CRUD для Admin) с полной DriverEditModal.
- **Форма E** — создание/редактирование отгрузки (модальное окно, реюз в обоих сценариях: прямой Create + Create-from-Plan-cell).
- **Модалка «Карточка водителя»** — открывается по клику на ФИО в таблице/строке отгрузки.

### 1.5 НЕ в скоупе MVP

| Печатные формы (реестр A4 портрет, сводка A4 ландшафт)                        | 3    |

### 1.6 Roadmap

- **Phase 1 (Core)** — этот документ. Текущий каркас + Главная (3 режима) + Sidebar + Role toggle + Drivers ref + Form E + Driver modal.
- **Phase 2 (Бизнес-разделы)** — Контракты, Аналитика, Логистика материалов, остальные справочники, модалка качества.
- **Phase 3 (Кросс-функциональное)** — Уведомления, Печать, Мобилка, Настройки, реальная авторизация.

### 3.3 RoleGate

Доступ к UI обёрнут в компонент:

```tsx
<RoleGate allow='admin'>
  <Button>Удалить</Button>
</RoleGate>
```

- `allow: Role | Role[]` — кому показывать.
- Если роль не подходит — не рендерит ничего (без fallback) или рендерит `fallback` если передан.
- Используется для кнопок, столбцов таблиц, gear-иконок, заголовков разделов sidebar, отметки прибытия.

## 4. Функциональные требования по экранам Core

### 4.1 Главная — Режим «Таблица»

_Источник: `reference/project/Wireframes.html` L547–960 (VariantA), L286–421 (Shipment row, ItemLine)._

#### 4.1.1 Группировка

- Двухуровневая: **Неделя → День → Отгрузки** (аккордеон).
- Заголовок недели: чёрная плашка (`.sk-week`, `#2a2a2a`/белый текст). Формат: «Неделя 17 · 21–26 апр 2025».
- Заголовок дня: тёмно-серая плашка (`.sk-day`, `#444`/белый текст). Формат: «ПН 21 апр · 4 отгрузки».
- Сворачивание: клик по заголовку недели или дня — collapse/expand с локальным `useState` (не персистится).

#### 4.1.2 Фильтры (top of section)

- **Период**: dropdown (сегодня / неделя / месяц / произвольный) — UI заглушка, по умолчанию «текущая неделя ±1». В Phase 1 — без логики, показываем все 3 недели из сидов.
- **Статус**: три чекбокса (`scheduled`, `sent`, `arrived`). Состояние в `useState`, не персистится. По умолчанию все три включены.
- Поиск, фильтр по поставщику/ТК — Phase 2 (см. §10).

#### 4.1.3 Строка отгрузки (Shipment row)

Колонки (горизонтально):

1. Дата отгрузки (`shipDate`) — формат «20 апр».
2. Дата поступления (`arrDate`) — формат «21 апр».
3. ФИО водителя — кликабельно → открывает модалку водителя (§4.4).
4. ТК — короткое название.
5. Статус — `<StatusChip status="..."/>` (см. §6).
6. **Action «✓ Отметить прибывшим»** — иконка справа от StatusChip. Показывается только для `scheduled` или `sent`, только для Admin (обёрнута в `<RoleGate allow="admin">`). Клик → confirm dialog «Отметить как прибыло?» → `ShipmentRepo.save({...s, status: 'arrived'})`.
7. № акта — Phase 2 (placeholder в Phase 1).
8. Σ вес — суммарный вес всех позиций машины, в кг.
9. Кол-во позиций — пилл «N позиций».
10. Опц. комментарий — italic курсив, оранжевый `#a06000`.
11. **Action «✏ Редактировать»** — открывает Form E в edit-mode. Доступно всем ролям (пользователь может править свои; правка ограничена статусом — см. §4.5.8).
12. **Action «🗑 Удалить»** — только Admin (`<RoleGate>`), confirm dialog.

#### 4.1.4 Позиция (ItemLine) внутри отгрузки

При раскрытии строки — список `ShipmentItem`:

- Цветная плашка сырья (фон `RAW_COLORS[rawId].bg` + dot).
- Название сырья.
- Вес: `{kg} кг` (моноширинный шрифт, JetBrains Mono).
- Поставщик: ФИО/название.
- Тара: отображение `tara[]` в формате «ящик × 583» или «бочка пластик. × 12, бочка металл. × 5». Если `tara=[]` (Phase 1 всегда; Phase 2 для сырья «без тары») — отображается «—». Read-only во всех сценариях.
- Опц. иконка качества (QualityBadge) — в Phase 1 показываем только визуально, без клика.

#### 4.1.5 Действия (top of page)

- Кнопка «+ Новая отгрузка» (top right) → открывает Form E в режиме create без префилла.

### 4.2 Главная — Режим «Heatmap»

_Источник: `reference/project/pivot-summary.jsx` L110–209 (PivotV2)._

#### 4.2.1 Оси

- **Строки**: основные + видимые сезонные `RawMaterial` (фильтрация по `WeekPlan.visibleRaws`).
- **Колонки**: 6 дней недели (ПН–СБ).
- **Ячейка**: суммарный вес (кг) по всем `ShipmentItem` с этим сырьём и этим днём (`arrDate.dayOfWeek`), включая все 3 статуса (scheduled + sent + arrived).

#### 4.2.2 Цветовая логика

- Базовый цвет = `RAW_COLORS[rawId].bg`.
- Интенсивность = `min(1, kg / maxKgInRow) * 0.85 + 0.15` (минимум 15% непрозрачности чтобы пустые ячейки оставались видимы) — функция `heatBg(rawId, v, maxV)` (см. `reference/project/pivot-summary.jsx` L116–127).
- Если ячейка пустая (`kg=0`) — фон `#f5f3ed` (серый).

#### 4.2.3 Контролы

- Переключатель внизу: «Сумма (кг) / Среднее за день / % от плана недели». В Phase 1 реализуем только **«Сумма (кг)»**, остальные кнопки disabled (визуально).
- Селектор недели: «← Пред / Сегодня / След →» (синхронизирован с PlanView, §4.3.2).

#### 4.2.4 Тоталы

- Колонка «Итого за неделю» по сырью (справа).
- Строка «Итого за день» по неделе (снизу).
- Угловая ячейка: сумма по всей неделе.

### 4.3 Главная — Режим «План»

\*Источник: `reference/project/plan-view.jsx` L470–1430. **Самый сложный экран Core.\***

#### 4.3.1 Структура

- Таблица: **8 основных овощей × 6 дней** (ПН–СБ). Сезонные показываются только если есть план > 0 или есть отгрузки.
- Ячейка содержит:
  - Числовой input (план в тоннах, integer, без десятых). Right-aligned.
  - Под input'ом: компьютед факт (тонны) и пилл состояния (см. §6.3).
  - `<ProgressBar pct={fact/plan*100} plan={plan} color={CELL_STYLES[state].bar}/>` — шкала 0–150%, маркер на 100%, штрих-перевыполнение поверх части >100%.
  - Если ячейка имеет shipments — иконка «N отгр.» внизу.
  - Клик по ячейке → открывает Form E с префиллом `{rawId, day, weekNum, year}` → **всегда новая Shipment** (§4.5.1).

#### 4.3.2 Навигация по неделям

- Три недели: `archive (-1)`, `current (0)`, `next (+1)`. Источник: `PlanRepo.byOffset(off)`.
- Контролы: «← Пред / Сегодня (выделено) / След →». Состояние в `useState`, синхронизируется в URL (`?week=18&year=2025`).
- **В Phase 1 архивная неделя одна — W16/2025 (offset=-1).** Флаг `archive=true` в сидах, не вычисление от today.
- Поведение на архивной неделе:
  - Все input'ы плана `readonly`.
  - Клик по ячейке → не открывает Form E (просто disabled state без тоста).
  - Пилл «🔒 Архив» в шапке Plan view над сеткой.
  - Gear (видимость) скрыт.

#### 4.3.3 Видимость овощей (gear popover)

- Иконка ⚙ справа над таблицей.
- Клик → popover со списком всех 14 овощей + чекбоксы.
- Хранение: `localStorage[wsm:v1:planVis:W{weekNum}_{year}]` = `string[]` (массив hidden raw ID).
- **Защита от скрытия (invariant)**: овощ нельзя скрыть, если на нём:
  - есть план > 0 в данной неделе, ИЛИ
  - есть хотя бы одна отгрузка с `ShipmentItem.rawId` за неделю.

## 5. Доменная модель

### 5.1 Принципы

- **Все ID — строки в формате nanoid()** или префиксированные (`drv_xxx`, `tk_xxx`, `sup_xxx`, `raw_xxx`).
- **Все enum-значения — английские строки** (`'scheduled' | 'sent' | 'arrived'`).

## 6. Бизнес-правила

### 6.1 Статусы отгрузки

Полный список (порядок жизненного цикла):

1. `scheduled` — «Запланировано». Машина заказана, выезд ожидается. **Дефолт при create.**
2. `sent` — «Отправлено». Машина выехала, в пути.
3. `arrived` — «Прибыло». Машина на заводе, разгружена.

Концепт «План» в Plan-mode — это **UI-состояние пустой ячейки** (план есть, факта нет). Не сохраняется в `Status` enum и не создаёт Shipment.

В Form E при создании доступны статусы `scheduled` и `sent` (две кнопки save).

- Из `scheduled` → переход в `sent` (Form E edit «Отправить») и далее в `arrived` (TableView action или Form E edit «Отметить прибывшим»).
- Из `sent` → переход в `arrived` (TableView action или Form E edit «Отметить прибывшим»).

### 6.2 Что считается «фактом» в Plan mode и Heatmap

**Все 3 статуса** (`scheduled`, `sent`, `arrived`) суммируются в факт ячейки.
_Источник: комментарий в `reference/project/pivot-summary.jsx` L4–6 и `plan-view.jsx` L4–7._

> Обоснование: heatmap и план показывают «что ожидается + что уже пришло» единым потоком, чтобы планёр видел, заполняется ли план или нет.

### 6.3 Цветовая логика ячейки плана

См. §4.3.5. ЗАФИКСИРОВАННОЕ ПРАВИЛО — не редизайнить. Шесть состояний.

### 6.4 Защита от скрытия овоща

См. инвариант I10. Если пользователь пытается снять чекбокс с защищённого овоща — чекбокс не реагирует, показывается тултип.

### 6.5 Архивная неделя

- `WeekPlan.archive === true` → весь экран Plan для этой недели рендерится без редактируемых input'ов.
- Кнопки «+ Новая отгрузка», edit shipment, mark arrived — disabled или скрыты в контексте архивной недели.
- Пилл «🔒 Архив» в шапке Plan view над сеткой.
- В Phase 1 архивная неделя ровно одна — W16/2025 (offset=-1 в сидах).

### 6.6 Дефолтный статус новой отгрузки

- При нажатии «Сохранить как запланировано» → `status='scheduled'`.
- При нажатии «✓ Сохранить и отправить» → `status='sent'`.
- В Plan-mode префилле: тот же выбор (default-action — `scheduled`).

### 6.7 Расчёт даты прибытия

- При вводе `shipDate` без `arrDate`: автоустанавливаем `arrDate = shipDate + 1 day`.
- Пользователь может изменить вручную (но ≥ shipDate).
- В Plan-mode `arrDate` префилл и locked (= день ячейки).

### 6.8 ТК автозаполнение

- При выборе `Driver` в Form E: `tkId` автоматически выставляется в `driver.tkId`.
- ТК становится readonly (но видим). Если пользователь меняет водителя — ТК обновляется.
- В Edit-mode: смена водителя пересчитывает ТК автоматически (не snapshot).

### 6.9 Удаление с FK

- `DriverRepo.delete(id)`: перед удалением проверяем `ShipmentRepo.countByDriver(id)`. Если > 0 — иконка 🗑 в UI disabled с tooltip; API-вызов всё равно бросает `CascadeRestrictError('Невозможно удалить: используется в N отгрузках')` для защиты от обхода.
- В Phase 1 это единственный случай каскада, потому что только Drivers редактируется.

### 6.10 Валидация на save (Form E)

Список см. §4.5.6. Реализация: pure-функция `validateShipment(s: Shipment): ValidationError[]` в `src/lib/validation.ts`. Возвращает массив ошибок (пустой = валидно). UI показывает их inline под полем.

### 6.11 Phone storage convention

Все телефонные номера в БД (LocalStorage сейчас, API позже) — **строго E.164 RU формат**: `^\+7\d{10}$`. UI выполняет форматирование на лету через хелперы.

**Хранение:** `+79012345678`
**Отображение:** `+7 (901) 234-56-78`
**tel: URI** (Phase 3 mobile): `tel:+79012345678`

Хелперы в `src/lib/phone.ts`:

- `normalizePhoneRu(input: string): string | null` — принимает свободный ввод (с пробелами, скобками, дефисами, без `+7`, с `8` в начале и т.д.), возвращает E.164 или `null` если не парсится.
- `formatPhoneRu(e164: string): string` — `+79012345678` → `+7 (901) 234-56-78`.
- `phoneTelUri(e164: string): string` — `+79012345678` → `tel:+79012345678`.

Компонент `<PhoneField>` (`src/components/atoms/PhoneField.tsx`, ~30 строк, без внешних библиотек масок):

- Принимает свободный ввод.
- На `onBlur` — нормализация через `normalizePhoneRu`. Если null — показывает ошибку и блокирует save родительской формы.
- Отображает форматированный вариант в read-state.

В `validateDriver` обязательная проверка `^\+7\d{10}$` на сохранённом значении.

В `DriverModal`: кнопка copy копирует `formatPhoneRu(phone)` (форматированную строку).

### 6.12 Автогенерация тары при отгрузке сырья (Phase 2)

При создании или редактировании `ShipmentItem` в Phase 2 система автоматически рассчитывает `tara: TaraEntry[]`:

1. Если `raw.allowedTara === []` → `tara = []` (без тары; огурцы, перец острый, перец сладкий).
2. Иначе ищется набор `TaraLoadNorm` по `(supplierId, rawId)`.
   - 0 норм → ошибка валидации «Не настроен норматив тары для фермера X и сырья Y».
   - 1 норма → `taraTypeId` берётся из неё автоматически.
   - ≥2 норм → Form E показывает select типа тары (значения = `allowedTara`); пользователь выбирает; в норме ищется выбранный `taraTypeId`.
3. Расчёт количества: `count = Math.round(kg / norm.fullTruckKg * norm.fullTruckCount)`.
4. Результат: `tara = [{ taraTypeId, count }]`.
5. Пользователь может скорректировать `count` вручную (например, при неполной загрузке). Тип тары не корректируется.

**Пример 1** (Цой, черри, ящики): норматив `fullTruckKg=18000`, `fullTruckCount=1050`. Отгрузка 10000 кг → `count = round(10000/18000 * 1050) = 583` ящика. `tara = [{ taraTypeId: 'tara_veg_box', count: 583 }]`.

**Пример 2** (Мищенко, патиссоны, железо): норматив `fullTruckKg=12000`, `fullTruckCount=60`. Отгрузка 8000 кг → `count = round(8000/12000 * 60) = 40` железных бочек. `tara = [{ taraTypeId: 'tara_metal_drum', count: 40 }]`.

**В Phase 1 это правило не реализуется.** `ShipmentItem.tara` всегда `[]`. Типы и инфраструктура подготовлены.

## 8. Архитектурные решения

### 8.1 Repository pattern

```ts
// src/repos/types.ts
export interface CrudRepo<T extends { id: string }, Q = void> {
  list(query?: Q): Promise<T[]>; // async для бизнес-логики и будущего API
  listSync(query?: Q): T[]; // sync snapshot из in-memory cache, для useSyncExternalStore
  get(id: string): Promise<T | null>;
  save(item: T): Promise<void>; // create или update
  delete(id: string): Promise<void>;
  subscribe(listener: () => void): () => void; // observer, см. §8.3
}

// Конкретные сабтипы с своими Q
export interface DriverQuery {
  search?: string;
  tkId?: string;
}
export type DriverRepo = CrudRepo<Driver, DriverQuery>;

export interface ShipmentQuery {
  weekNum?: number;
  year?: number;
  status?: Status[];
  driverId?: string;
}
export interface ShipmentRepo extends CrudRepo<Shipment, ShipmentQuery> {
  countByDriver(driverId: string): Promise<number>;
  // ... другие aggregation-методы по мере надобности
}

export interface PlanRepo extends CrudRepo<WeekPlan, void> {
  byOffset(offset: -1 | 0 | 1): Promise<WeekPlan>;
}
```

### 8.3 Observer для реактивности

- `subscribe(listener)` → возвращает unsubscribe.
- `save`/`delete` после записи в LS вызывают `notifyAll()` → все listener'ы выполняются.
- **Phase 1: notify all listeners on any save/delete (broad notify).** Phase 2: гранулярная инвалидация по entity+query при необходимости (см. §10 Out of scope). Зафиксировать комментарием в коде `LocalStorageRepo`.
- React-обёртка через `useSyncExternalStore`:
  ```ts
  function useShipments(query: ShipmentQuery) {
    const repo = useShipmentRepo();
    return useSyncExternalStore(
      (cb) => repo.subscribe(cb),
      () => repo.listSync(query),
    );
  }
  ```
- Это даёт **автоматическую реактивность**: PlanView сразу пересчитает факт при сохранении новой Shipment, без manual refresh.
